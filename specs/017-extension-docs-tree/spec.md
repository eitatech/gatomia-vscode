# Feature Specification: Dynamic Extension Document Display in Spec Explorer

**Feature Branch**: `017-extension-docs-tree`  
**Created**: 2026-04-13  
**Status**: Draft  
**Input**: User description: "The Spec Kit now supports extensions. These extensions can generate other markdown documents in the spec directory. We need to ensure that these generated assets are visible in our document tree, just as with spec, plan, checklist, etc. documents. We need to modify our code so that the extension reads these documents dynamically and displays them in the view."

## Clarifications

### Session 2026-04-13

- Q: Should this feature apply to both SpecKit and OpenSpec systems? → A: SpecKit only.
- Q: Should nested subdirectories within extension subfolders be shown? → A: Fully recursive (show entire directory tree regardless of depth).
- Q: Should extension-generated documents be visually distinguishable from standard spec artifacts? → A: Yes, use a distinct icon (e.g., `extensions` or `puzzle-piece` theme icon).
- Q: What is the acceptable maximum latency for the file watcher to trigger a tree refresh? → A: 2 seconds with debounce.
- Q: How should extension subfolder display labels be derived? → A: From folder name only (kebab-case to title case), no dependency on `extension.yml`.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - View Extension-Generated Documents in Spec Explorer (Priority: P1)

As a developer using SpecKit with extensions installed, I want any extra markdown files present in a spec directory (beyond the standard spec, plan, tasks, etc.) to automatically appear in the Spec Explorer tree view so that I can navigate to them without leaving the IDE.

**Why this priority**: This is the core value of the feature. Without it, extension-generated files are invisible in the tree and users must manually browse the filesystem to find them, breaking the single-pane workflow that the Spec Explorer provides.

**Independent Test**: Can be fully tested by placing an extra `.md` file (e.g., `retrospective.md`) into any spec directory and verifying it appears as a child node in the Spec Explorer tree under that spec.

**Acceptance Scenarios**:

1. **Given** a spec directory `specs/016-multi-provider-agents/` contains `retrospective.md` (generated by the retrospective extension), **When** the Spec Explorer tree is expanded for that spec, **Then** a "Retrospective" node appears alongside the standard Spec, Plan, Tasks nodes.
2. **Given** a spec directory contains multiple extra markdown files (e.g., `requirements-spec.md`, `system-design.md`, `acceptance-test-plan.md`), **When** the tree is expanded, **Then** all extra files appear as individual leaf nodes with appropriate labels and icons.
3. **Given** a spec directory contains only the standard files (spec.md, plan.md, tasks.md), **When** the tree is expanded, **Then** no extra nodes appear and the tree looks identical to current behavior.

---

### User Story 2 - Extension Documents Grouped in Subfolders (Priority: P2)

As a developer whose extensions generate multiple related documents into a subfolder (e.g., a `v-model/` directory inside the spec), I want those subfolders to appear as collapsible groups in the Spec Explorer so I can browse them hierarchically.

**Why this priority**: Extensions like the V-Model pack can generate many documents. Displaying them flat would clutter the tree. Grouping by subfolder preserves organization and mirrors the filesystem layout.

**Independent Test**: Can be tested by creating a subfolder (e.g., `specs/016-multi-provider-agents/v-model/`) with markdown files inside, and verifying it appears as a collapsible folder node in the tree.

**Acceptance Scenarios**:

1. **Given** a spec directory contains a subfolder `v-model/` with `requirements-spec.md` and `system-design.md` inside, **When** the tree is expanded for that spec, **Then** a "V Model" collapsible folder node appears, and expanding it shows the two documents.
2. **Given** a subfolder is empty, **When** the tree is rendered, **Then** the empty subfolder does not appear in the tree.
3. **Given** a subfolder is one of the already-handled folders (checklists, contracts), **When** the tree is rendered, **Then** it continues to use the existing specialized rendering (with status icons, etc.) rather than the generic folder rendering.

---

### User Story 3 - Dynamic Refresh on File System Changes (Priority: P3)

As a developer who just ran an extension command that generated new documents, I want the Spec Explorer to reflect the new files without requiring a manual refresh or IDE restart.

**Why this priority**: Important for a smooth workflow, but the existing manual refresh button already provides a workaround. Automatic detection is a quality-of-life improvement.

**Independent Test**: Can be tested by starting a file watcher, creating a new `.md` file in a spec directory, and verifying the tree updates within a reasonable time.

**Acceptance Scenarios**:

1. **Given** the Spec Explorer is open and showing a spec, **When** a new markdown file is added to that spec's directory by an extension command, **Then** the tree updates to show the new file within 2 seconds (debounced).
2. **Given** the Spec Explorer is open, **When** an extension-generated file is deleted, **Then** the corresponding tree node disappears after the next refresh cycle.

---

### Edge Cases

- What happens when a non-markdown file (e.g., `.yml`, `.json`, `.png`) exists in the spec directory? It should be ignored since the tree only displays markdown documents.
- What happens when a deeply nested subfolder structure exists (e.g., `v-model/tests/unit/`)? The full directory tree is displayed recursively regardless of depth.
- What happens when a subfolder has the same name as a known document type (e.g., a folder called `spec/`)? It should be treated as a generic subfolder, not confused with `spec.md`.
- What happens when file names contain special characters or very long names? The display label should be derived by removing the `.md` extension and formatting kebab-case to sentence case (capitalize first letter, replace hyphens with spaces), matching existing behavior.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The Spec Explorer MUST display any `.md` file found in a SpecKit spec directory (`specs/NNN-feature/`) that is not already handled by the existing known-document logic (spec, plan, research, data-model, quickstart, tasks, checklists, contracts). OpenSpec directories are out of scope.
- **FR-002**: Extra markdown files MUST appear as leaf nodes with a distinct icon (e.g., `extensions` or `puzzle-piece` theme icon) and a human-readable label derived from the filename (kebab-case to sentence case: capitalize first letter, replace hyphens with spaces, strip `.md` extension). This icon MUST differ from icons used for standard spec artifacts to provide visual distinction.
- **FR-003**: Unknown subfolders within a spec directory MUST appear as collapsible folder nodes in the tree, with their contents displayed fully recursively (nested subdirectories shown as collapsible children, and so on at any depth). Display labels for folders MUST be derived from the folder name using kebab-case to sentence case formatting (capitalize first letter, replace hyphens with spaces).
- **FR-004**: Known subfolders (checklists, contracts) MUST continue to use their existing specialized rendering and MUST NOT be duplicated by the generic folder logic.
- **FR-005**: The tree MUST preserve the existing ordering: known documents first (in their current order), followed by extra documents and subfolders sorted alphabetically.
- **FR-006**: Clicking an extra document node MUST open the corresponding file in the VS Code editor, using the existing `gatomia.spec.open` command.
- **FR-007**: The display MUST be fully dynamic; no hardcoded list of extension-generated filenames may be used. Any `.md` file or subfolder present at the time of tree rendering MUST be shown.
- **FR-008**: Non-markdown files in the spec directory MUST be ignored and not displayed in the tree.
- **FR-009**: Empty subfolders MUST NOT appear in the tree.

### Key Entities

- **Extra Document**: A markdown file in a spec directory that is not one of the predefined known types (spec, plan, research, data-model, quickstart, tasks) and not inside a known subfolder (checklists, contracts). Represents an extension-generated artifact.
- **Extension Subfolder**: A directory inside a spec folder that is not one of the known subfolder types (checklists, contracts). May contain extension-generated markdown documents grouped by extension or purpose.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of markdown files present in any spec directory are visible in the Spec Explorer tree within one refresh cycle.
- **SC-002**: Users can open any extension-generated document from the tree in a single click, with no additional navigation required.
- **SC-003**: Existing tree behavior for known document types (spec, plan, tasks, checklists, contracts) remains identical; no regression in current functionality.
- **SC-004**: The tree correctly handles specs with zero, one, or many extension-generated documents without layout issues or performance degradation.

## Assumptions

- Extensions generate their output as `.md` files directly in the spec directory or in subfolders within it.
- The `extension.yml` files in `.specify/extensions/` define what commands extensions provide, but the tree display does not depend on reading these files. Instead, it dynamically scans the filesystem.
- The existing `getSpecKitFeatureFiles` method in `spec-kit-adapter.ts` is the primary integration point for collecting files to display.
- File system watchers (if implemented) can rely on VS Code's `workspace.createFileSystemWatcher` API.
